# Feature Specification: STT 기반 지식 DB 자동 구축 및 AI 에이전트 활용

**Feature Branch**: `001-stt-knowledge-db`
**Created**: 2026-03-28
**Status**: Draft
**Input**: User description: "자동으로 지식 DB를 구축하고 이를 AI 에이전트가 사용할 수 있도록 제공하는 프로그램. 마이크 활성화 → VAD → STT → 마크다운 지식 DB 저장. AI 에이전트가 축적된 지식을 활용하여 암묵지 기반 응답 제공."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 음성으로 지식 자동 수집 (Priority: P1)

사용자가 프로그램을 실행하면 마이크가 활성화되고, 말하는 내용이 자동으로 텍스트로 변환되어 지식 DB에 저장된다. 사용자는 별도의 타이핑이나 수동 작업 없이 일상적인 대화나 설명을 통해 지식을 축적할 수 있다.

**Why this priority**: 암묵지를 최소화하는 것이 AI Native한 방법이다. 이것이 있어야 매번 AI에게 프롬프트로 맥락을 설명하지 않고도 필요한 지식을 제공할 수 있다. 구두로 하는 논의들이 많이 있기 때문에 이를 AI가 향후 접근할 수 있는 방법을 제시해야한다.

**Independent Test**: 프로그램을 실행하고 음성으로 말한 뒤, 지식 DB 디렉토리에 해당 내용이 마크다운 파일로 저장되었는지 확인하여 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 프로그램이 실행된 상태, **When** 사용자가 말을 시작하면, **Then** VAD가 음성을 감지하고 STT가 시작된다
2. **Given** VAD가 음성을 감지한 상태, **When** 사용자가 말을 끝내면(침묵 구간 감지), **Then** STT 결과물이 마크다운 파일로 지식 DB 디렉토리에 저장된다
3. **Given** 프로그램이 실행된 상태, **When** 주변 소음만 있고 사용자가 말하지 않으면, **Then** VAD가 음성으로 판단하지 않아 STT가 수행되지 않는다
4. **Given** STT 결과물이 생성되면, **When** 마크다운 파일이 저장될 때, **Then** 파일 상단에 지식 요약 설명이 있고 하단에 상세 내용이 포함된다

---

### User Story 2 - 지식 DB를 활용한 AI 에이전트 응답 (Priority: P2)

사용자가 AI 에이전트에게 질문하면, 에이전트가 축적된 지식 DB(마크다운 파일들)를 참조하여 사용자가 프롬프트에 명시하지 않은 맥락과 암묵지까지 반영한 더 정확한 답변을 제공한다.

**Why this priority**: 지식 DB가 구축된 후에야 활용할 수 있는 기능이며, 이 기능이 있어야 전체 시스템의 가치가 완성된다.

**Independent Test**: 사전에 지식 DB에 특정 도메인 지식을 저장해둔 상태에서, 해당 맥락이 필요한 질문을 AI 에이전트에게 했을 때 지식 DB 내용을 반영한 답변이 돌아오는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 지식 DB에 특정 주제의 마크다운 파일들이 저장된 상태, **When** 사용자가 해당 주제에 관한 질문을 하면, **Then** AI 에이전트가 관련 지식을 참조하여 맥락이 반영된 답변을 제공한다
2. **Given** 지식 DB가 비어 있는 상태, **When** 사용자가 질문을 하면, **Then** AI 에이전트가 일반적인 답변을 제공하되, 지식 DB가 비어 있음을 안내한다
3. **Given** 지식 DB에 여러 주제의 지식이 축적된 상태, **When** 사용자가 복합적인 질문을 하면, **Then** AI 에이전트가 관련된 여러 지식 파일을 종합하여 답변한다

---

### User Story 3 - 지식 DB 구조 탐색 및 확인 (Priority: P3)

사용자가 축적된 지식 DB의 구조와 내용을 확인할 수 있다. 디렉토리 구조로 정리된 마크다운 파일들을 통해 어떤 지식이 저장되어 있는지 파악하고, 필요시 수동으로 편집할 수 있다.

**Why this priority**: 지식 DB의 투명성과 사용자 신뢰를 위해 필요하지만, 핵심 흐름(수집/활용)이 먼저 동작해야 의미가 있다.

**Independent Test**: 지식이 여러 건 축적된 상태에서 디렉토리 구조를 확인하고, 각 마크다운 파일의 상단 설명이 내용과 일치하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 여러 건의 지식이 저장된 상태, **When** 사용자가 지식 DB 디렉토리를 열면, **Then** 주제별로 분류된 디렉토리 구조와 마크다운 파일 목록을 확인할 수 있다
2. **Given** 마크다운 지식 파일이 있을 때, **When** 파일을 열면, **Then** 상단에 지식 요약/설명이 있고, 하단에 상세 원문 내용이 있다
3. **Given** 사용자가 마크다운 파일을 직접 수정하면, **When** AI 에이전트가 해당 지식을 참조할 때, **Then** 수정된 내용이 반영된다

---

### Edge Cases

- 매우 짧은 발화는 최소 발화 길이 임계값(기본 3초)으로 필터링하여 버린다. 임계값은 설정으로 조정 가능하다
- STT 인식 정확도가 낮은 경우(소음 환경, 불명확한 발음) 처리 방식
- 동시에 여러 사람이 말하는 경우 처리 방식
- 카테고리 오분류 시 사용자가 파일을 올바른 디렉토리로 직접 이동하고 frontmatter category를 수동 수정하여 보정한다
- 지식 DB에 매우 많은 파일이 쌓였을 때 AI 에이전트의 참조 성능
- 마이크 권한이 거부되었을 때의 사용자 안내
- STT 서비스 연결 실패 시 대응 방식
- Claude Code CLI 호출 실패(네트워크 오류 등) 시 해당 STT 결과의 파일 저장을 건너뛰고 로그에만 기록한다 (데이터 유실 허용)

## Interface Strategy

### Architecture: 코어 라이브러리 + 다중 진입점

코어 로직(VAD, STT, 저장, 검색)은 Go 패키지(`pkg/`)로 구현하고, 각 인터페이스는 이 패키지를 호출하는 얇은 진입점으로 구성한다. 이를 통해 인터페이스를 추가하거나 교체해도 비즈니스 로직을 변경할 필요가 없다.

```
sttdb/
├── pkg/          ← 코어 라이브러리 (VAD, STT, 저장, 검색)
├── cmd/cli/      ← CLI 진입점
├── cmd/mcp/      ← MCP 서버 진입점
└── cmd/app/      ← (Phase 2) 데스크탑 앱 진입점
```

### Phase 1 (현재 범위): CLI + MCP 서버

| 대상 | 인터페이스 | 용도 |
|------|-----------|------|
| 사람 (개발자/PO/디자이너) | **CLI + 파일시스템** | 음성 수집 제어, 지식 조회/검색, 마크다운 직접 편집 |
| AI 에이전트 | **MCP 서버** | 지식 검색/조회를 MCP tool로 노출 |

**CLI 주요 명령어**:
- `sttdb start` / `sttdb stop` / `sttdb status` — 음성 수집 데몬 제어
- `sttdb search <query>` — 지식 검색
- `sttdb list` — 지식 목록 조회

**MCP tool**:
- `search_knowledge(query, category?, include_chitchat?)` — 키워드 매칭 기반 지식 검색 (카테고리 지정 시 해당 범위 우선 탐색, `잡담/` 카테고리는 기본 제외, `include_chitchat: true` 또는 `category: 잡담` 시 포함)
- `list_knowledge(category?, date_range?)` — 지식 목록 조회
- `get_knowledge(id)` — 특정 지식 상세 조회

### Phase 2 (향후): 데스크탑 앱

- 동일한 `pkg/` 코어 라이브러리 위에 데스크탑 UI를 추가한다
- CLI와 MCP 서버는 계속 병행 제공한다
- 데스크탑 프레임워크 선정은 Phase 2 시작 시점에 결정한다 (YAGNI)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 프로그램 실행 시 마이크를 활성화하고 오디오 입력을 수신해야 한다
- **FR-002**: 시스템은 VAD(Voice Activity Detection)를 통해 사람의 음성이 감지될 때만 STT를 수행해야 한다. 최소 발화 길이(기본 3초) 미만의 음성 구간은 무시하며, 임계값은 설정 파일(`~/.sttdb/config.yaml`)로 조정 가능하다
- **FR-003**: 시스템은 음성 구간이 끝난 후(침묵 감지) 로컬 Whisper `base` 모델(기본값, 설정으로 변경 가능)을 통해 STT 결과물을 생성해야 한다
- **FR-004**: 시스템은 STT 결과물을 마크다운 파일 형태로 지식 DB 디렉토리(`~/.sttdb`)에 저장해야 한다
- **FR-005**: 각 지식 마크다운 파일은 YAML frontmatter(`---` 블록)에 메타데이터(title, category, created_at)를 포함하고, 본문에 지식 요약 설명과 상세 원문 내용을 포함해야 한다. 요약 설명과 메타데이터는 Claude Code CLI를 활용하여 자동 생성한다
- **FR-006**: 시스템은 지식 파일을 최대 2단계(대분류/소분류) 카테고리 계층 디렉토리 구조로 분류하여 저장해야 한다. Claude Code CLI를 활용하여 STT 결과물의 주제를 자동 분류하고, 해당 카테고리 디렉토리에 파일을 배치한다 (예: `개발/에러처리/20260328-143052.md`). 카테고리는 사전 정의 없이 Claude가 기존 디렉토리 구조를 참조하여 최적의 카테고리를 자동으로 판단·생성한다. `잡담/`은 업무 무관 발화를 위한 카테고리로, 다른 업무 카테고리와 동일 레벨에 병렬 배치한다
- **FR-007**: AI 에이전트는 MCP 서버를 통해 지식 DB의 마크다운 파일들을 참조하여 사용자 질문에 답변해야 한다. MCP tool은 키워드 매칭과 카테고리 탐색 등 원시 접근 수단을 제공하고, 검색 전략(범위 좁히기, 키워드 변경, 관련성 판단)은 AI 에이전트가 수행한다. 별도 검색 엔진은 사용하지 않는다. `잡담/` 카테고리는 기본 검색 범위에서 제외되며, `category: 잡담`을 명시적으로 지정할 때만 검색에 포함된다
- **FR-008**: AI 에이전트는 사용자가 명시하지 않은 맥락도 지식 DB에서 찾아 답변에 반영해야 한다
- **FR-009**: 시스템은 마이크 권한 미허용 시 사용자에게 명확한 안내를 제공해야 한다
- **FR-010**: 시스템은 실행 중 지속적으로 음성을 감지하며, 사용자가 명시적으로 종료할 때까지 동작해야 한다
- **FR-011**: 시스템은 CLI를 통해 음성 수집 데몬의 시작/중지/상태 확인을 제어할 수 있어야 한다. 데몬은 PID 파일(`~/.sttdb/sttdb.pid`)로 관리하며, stale PID 감지 시 자동 정리한다
- **FR-012**: 시스템은 CLI를 통해 지식 DB의 검색 및 목록 조회를 제공해야 한다
- **FR-013**: 시스템은 MCP 서버를 통해 AI 에이전트에게 지식 검색/조회/상세 보기 tool을 제공해야 한다

### Key Entities

- **Knowledge Entry (지식 항목)**: 한 번의 음성 발화에서 생성된 지식 단위. YAML frontmatter에 title, category, created_at 메타데이터를 포함하고, 본문에 요약 설명과 상세 원문 내용을 포함한다
- **Knowledge Base (지식 DB)**: `~/.sttdb` 디렉토리 하위에 주제 카테고리 계층 디렉토리로 구조화된 마크다운 파일들의 모음. Claude Code CLI가 STT 결과를 분석하여 주제를 자동 분류하고, 계층적 디렉토리에 배치한다 (예: `~/.sttdb/개발/에러처리/`, `~/.sttdb/디자인/UX패턴/`, `~/.sttdb/잡담/`). 업무 무관 발화는 `잡담/` 카테고리로 분류되어 업무 지식과 분리된다
- **Audio Segment (음성 구간)**: VAD에 의해 감지된 하나의 연속 음성 구간. STT 처리의 입력 단위이다

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 프로그램 실행 후 3초 이내에 마이크가 활성화되어 음성 수집이 시작된다
- **SC-002**: 사용자가 말을 끝낸 후 5초 이내에 해당 내용이 마크다운 파일로 저장된다
- **SC-003**: 조용한 환경에서 일반적인 발화의 90% 이상이 의미 있는 텍스트로 변환된다
- **SC-004**: AI 에이전트가 지식 DB에 존재하는 주제에 대해 질문받았을 때, 관련 지식을 80% 이상의 확률로 참조한다
- **SC-005**: 지식 DB에 100개 이상의 항목이 저장되어도 AI 에이전트 응답 시간이 30초를 초과하지 않는다
- **SC-006**: 사용자가 별도의 설정 없이 프로그램을 실행하는 것만으로 지식 수집을 시작할 수 있다

## Clarifications

### Session 2026-03-28

- Q: STT 서비스 구현 방식 (Cloud API vs Local Whisper vs Hybrid)? → A: Local Whisper (로컬에서 OpenAI Whisper 모델 실행)
- Q: 지식 파일 요약 설명 자동 생성 방식? → A: Claude Code CLI를 활용하여 STT 결과물의 요약 제목/설명 생성
- Q: 지식 파일 이름 규칙과 디렉토리 분류 기준? → A: 주제 카테고리 계층 디렉토리 구조 (Claude Code CLI로 주제 자동 분류)
- Q: 짧은 발화(1-2초 미만) 처리 방식? → A: 최소 발화 길이 임계값 설정 (기본 3초 미만 버림, 설정으로 조정 가능)
- Q: MCP 서버 지식 검색 방식? → A: MCP tool은 키워드 매칭·카테고리 탐색 등 원시 접근 수단 제공, 검색 전략은 AI 에이전트가 수행 (별도 검색 엔진 없음)
- Q: 지식 DB 파일 저장 경로? → A: `~/.sttdb` 디렉토리
- Q: Whisper 모델 크기 기본값? → A: `base` (74M, 속도/정확도 균형, 설정으로 변경 가능)
- Q: Claude Code CLI 호출 실패 시 처리 방식? → A: 파일 저장을 건너뛰고 로그에만 기록 (데이터 유실 허용)
- Q: 마크다운 지식 파일 메타데이터 형식? → A: YAML frontmatter (`---` 블록에 title, category, created_at 등)
- Q: 설정 파일 위치 및 형식? → A: `~/.sttdb/config.yaml` (지식 DB 디렉토리 내 YAML 파일)
- Q: 데몬 프로세스 관리 방식? → A: PID 파일(`~/.sttdb/sttdb.pid`) 기반, 크래시 시 stale PID 감지 후 정리
- Q: 최상위 카테고리 구조 (업무 vs 잡담 분리 방식)? → A: 병렬 카테고리 — `잡담/`을 다른 업무 카테고리와 동일 레벨에 배치 (예: `개발/에러처리/`, `디자인/`, `잡담/`)
- Q: 카테고리 계층 깊이 제한? → A: 최대 2단계 (대분류/소분류, 예: `개발/에러처리/`, `잡담/일상/`)
- Q: 카테고리 목록 관리 방식? → A: 완전 자동 — 사전 정의 없이 Claude가 매번 기존 카테고리 구조를 참조하여 최적의 카테고리를 판단·생성·배치
- Q: 잡담 오분류 시 사용자 보정 방식? → A: 파일 직접 이동 — 사용자가 파일을 올바른 카테고리 디렉토리로 이동하고 frontmatter category를 수동 수정
- Q: AI 에이전트의 잡담 카테고리 검색 제외 여부? → A: 기본 제외 — 잡담 카테고리는 기본 검색에서 제외, `category: 잡담` 명시 시에만 포함

## Assumptions

- 사용자의 기기에 마이크가 내장되어 있거나 연결되어 있다
- 사용자는 주로 한국어로 발화하며, 단일 언어 STT로 충분하다
- STT는 로컬 Whisper `base` 모델(기본값)로 실행하며, 인터넷 연결 없이 동작한다 (오프라인 사용 가능). 모델 크기는 설정으로 변경 가능하다
- 지식 DB는 로컬 파일시스템(`~/.sttdb`)에 저장되며, 클라우드 동기화는 범위 밖이다
- AI 에이전트는 기존 LLM 서비스를 활용하며, 자체 모델 학습은 범위 밖이다
- 동시에 한 명의 사용자만 마이크를 사용한다 (다중 화자 분리는 범위 밖)
- 마크다운 파일의 요약 설명은 Claude Code CLI를 호출하여 STT 결과를 기반으로 자동 생성된다 (요약 생성 시 인터넷 연결 필요)
- 초기 버전에서는 지식 파일의 수동 편집/삭제 기능은 파일 시스템 직접 접근으로 대체한다
- 사용자 설정(Whisper 모델 크기, 최소 발화 길이 등)은 `~/.sttdb/config.yaml`에 저장한다. 설정 파일이 없으면 기본값으로 동작한다
- 코어 로직은 Go 패키지(`pkg/`)로 분리하며, CLI/MCP/데스크탑 앱 모두 이 패키지를 호출한다
- Phase 1에서는 CLI와 MCP 서버만 구현하며, 데스크탑 앱은 Phase 2 범위이다
- 대상 사용자는 개발자, PO, 디자이너이며, 모두 CLI 사용이 가능하다
